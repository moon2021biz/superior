🔍発見された課題（8件）

1
Step 1 が長大すぎる
深刻
5フェーズ（A〜E）を一気に実行すると出力が膨大になる。要約＋専門家5点＋コラム5点＋構成3点＋再構築版＋4つの確認問を1つのレスポンスで読みこなすのは認知負荷が高く、ユーザーが途中でスキップしがちになる。
影響：ユーザーが確認を飛ばして「D→OK」を押すだけになり、品質向上ループが機能しない
2
確認回数が多い（6回）
深刻
Step 1・2・2.5・3・4 で計6回ユーザー確認が発生する。1回の確認を経て次のステップに進む構造のため、セッションを跨がないと完結しないことが多い。実際にはStep 1の4問を一度に回答するのも難しい。
影響：1セッションでStep 4まで到達することが稀。途中で止まってハンドオフが残る
3
Step 2.5 のパスハードコード問題
深刻
Pythonコード内のダウンロードパスが古いセッションID great-vigilant-lovelace でハードコードされている。セッションを跨ぐたびにパスが変わるため、実行するとFileNotFoundErrorが確実に発生する。
影響：Step 2.5のPPTX画像抽出が100%エラーで失敗する（Step 8も同様）
4
Computer Use依存（3ステップ）
深刻
Step 2.5・6・8 がClaude in ChromeによるNotebookLM操作に依存している。Computer Useが「設定→デスクトップアプリ→コンピュータ使用」で有効化されていないと、これらのステップで完全に詰まる。有効化されていない場合の代替フローが定義されていない。
影響：Computer Use無効時はStep 2.5で停止→YouTube台本まで到達できない
5
NotebookLMノートブック肥大化
中程度
Step 6で毎回同じNotebookLMノートブックに5種類のソースを追加し続ける。回数を重ねるとソース数が増え続け、スライド生成の精度低下・動作遅延・関係のない過去テーマの混入リスクがある。ソースのクリーンアップ手順が定義されていない。
影響：10回以上使用後からスライドの品質が劣化する可能性
6
PPTX画像番号マッピングの不安定性
中程度
Step 2.5の画像抽出コードで image1.png=スライドA・image2.png=スライドB という前提でマッピングしている。しかしPPTX内のmediaフォルダはスライド順ではなく追加順で番号が振られるため、ロゴ等の埋め込み画像が含まれると番号がずれる。
影響：誤ったスライドが保存されることがある（サムネイルにロゴ画像が来る等）
7
YouTube・TikTokの並列化が未対応
軽微
NOTE記事（Step 2）と統一文言（Step 3）が確定すれば、YouTube台本（Step 4）とTikTok投稿文（Step 5）は独立して生成できる。しかし現在は直列設計のため、Step 4承認を待ってからStep 5が開始される。
影響：Step 4・5を並列化すれば所要時間を20〜30%短縮できる機会が失われている
8
エラー時のフォールバックが未定義
中程度
NotebookLMが503エラー・生成タイムアウト・ログイン切れ等で失敗した場合の対処が「代替手段は禁止（SKILL.md記載）」のみで、ユーザーへの報告内容・次のアクションが定義されていない。CLAUDE.mdのLEVEL3フォールバックへの接続が不明確。
影響：NotebookLMエラー時にスキルが無言で止まるか、誤った代替処理が動く

💡改善提案（7件）

✂️
提案①：Step 1 を3段階に分割する
課題 #1 #2 に対応
実装: 容易 効果: 高
現在の5フェーズ一括実行を、以下の3段階に分割して確認を分散させる。
  • Step 1a: フェーズA（詳細要約）のみ実行 → 「テーマ・ターゲット・メインメッセージ・フック案・有料コンテンツ候補」を確認。少ない情報量でスキャンしやすい。
  • Step 1b: フェーズB（専門家レビュー）+ フェーズC（コラムリスト介入）を実行 → 「どの指摘を採用するか」番号選択で回答。
  • Step 1c: フェーズD+E（構成改善 + 再構築版）を実行 → 最終確認してStep 2へ。
ユーザーは1回あたり少ない量を確認するだけで済み、スキップされにくくなる。
BEFORE
1回で5フェーズ全出力 → 4問同時回答 → 認知負荷MAX
AFTER
1a確認（1問）→ 1b確認（1問）→ 1c確認（1問）→ 段階的

提案②：クイックモードを追加する
課題 #2 に対応
実装: 容易 効果: 高
スキル起動時に「通常モード（デフォルト）」か「クイックモード」を選択させる。
  • 通常モード: 現在の全ステップ（レビュー・コラム介入・構成改善を含む完全版）
  • クイックモード: Step 1を「フェーズA（要約）のみ」に短縮→確認1回→Step 2→Step 3→Step 5（YouTube台本はスキップ）の最短4ステップ完結
週複数本の投稿を量産する場合や、テーマが決まっていてレビューが不要な場合にクイックモードが有効。
BEFORE
全8ステップ固定 / 確認6回 / 完了まで複数セッション
AFTER（クイックモード）
4ステップ / 確認2回 / 1セッションで完結

提案③：パスをすべて動的生成に修正する
課題 #3 に対応（最優先修正）
実装: 容易 効果: 必須
Step 2.5のPythonコード内にある古いセッションIDを削除し、実行時に動的取得する方式に変更する。 SKILL.md内のコードを以下のように修正すること： 
# ❌ 現在（ハードコード）
downloads_dir = '/sessions/great-vigilant-lovelace/mnt/Downloads'

# ✅ 修正案（動的取得）
import glob, os
# Bashで事前取得: SESSION_MNT=$(ls /sessions/*/mnt/Downloads -d 2>/dev/null | head -1)
# または Pythonで検索
session_dirs = glob.glob('/sessions/*/mnt/Downloads')
downloads_dir = session_dirs[0] if session_dirs else None
if not downloads_dir:
    raise FileNotFoundError("Downloadsフォルダが見つかりません。フォルダアクセスを許可してください。")

# 出力先も同様に動的生成
mnt_root = os.path.dirname(os.path.dirname(downloads_dir))
output_dir = f'{mnt_root}/ai実験用/オギラボ/outputs'

提案⑤：NotebookLMソース管理（クリーンアップ手順追加）
課題 #5 に対応
実装: 中程度 効果: 中
Step 6の冒頭にクリーンアップ手順を追加する。毎回使うソースはノートブックに固定し、テーマごとに変わるソース（投稿文テキスト）のみ毎回差し替える運用に変更する。
  • 固定ソース（一度追加したら削除しない）: OGILABロゴ・キャラクター画像・TikTok URL・YouTube URL
  • 動的ソース（毎回差し替え）: 投稿文テキスト → 前回のソースを削除してから新規追加する
この変更でノートブックのソース数を5以下に保ち、生成精度を維持する。

提案⑥：PPTX画像抽出を「スライド順」で正確に行う
課題 #6 に対応
実装: 中程度 効果: 高
ppt/media/image1.png = スライド1という前提は間違いの場合がある。python-pptxを使って正しいスライド順で画像を抽出する方式に変更する。 
# ✅ python-pptx を使って正確にスライド順で画像抽出
from pptx import Presentation
from pptx.util import Inches
from PIL import Image
import zipfile, io

prs = Presentation(pptx_path)
slide_images = []

# 各スライドをPILイメージとしてレンダリング（python-pptxはレンダリング不可のため）
# → zipで直接 slide/slide{N}.xml を開き、対応するblipを取得する方式
with zipfile.ZipFile(pptx_path, 'r') as z:
    for i, slide in enumerate(prs.slides):
        slide_xml = f'ppt/slides/slide{i+1}.xml'
        # blip参照からmediaファイルパスを解決 → 正確なスライド→画像マッピング
        rels_xml = f'ppt/slides/_rels/slide{i+1}.xml.rels'
        # 実装詳細はスキル内コードとして展開
        print(f"スライド{i+1}: {slide_xml}")
